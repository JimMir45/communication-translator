## 测试用例 1：产品 → 开发

**接口**：`POST /api/translate`
**方向**：`pm_to_dev`
**状态**：✅ 通过

### 输入

```
我们需要一个智能推荐功能，提升用户停留时长。
希望根据用户历史行为，给他推荐他感兴趣的内容，让用户在平台上多待一会。
```

### 翻译输出

## 需求本质（技术视角重述）

基于用户行为日志，构建一套内容召回 → 粗排 → 精排 → 重排的推荐链路，通过个性化排序替代现有的时间线/人工运营排序，最终目标是**提升用户单次会话时长（Session Duration）和内容消费深度（如翻页数/播放完成率）**。

核心技术问题是：**在冷启动、实时性、工程复杂度三者之间找到当前阶段合理的平衡点。**

---

## 技术实现方案

### 推荐方案选型（按复杂度递进）

| 方案 | 适用阶段 | 核心逻辑 | 实时性 | 工程复杂度 |
|---|---|---|---|---|
| **方案A：协同过滤 + 规则重排** | 冷启动期（DAU < 10万） | UserCF/ItemCF，相似用户/内容召回 | T+1 离线 | 低 |
| **方案B：双塔模型召回 + LightGBM 精排** | 成长期（DAU 10万~500万） | Embedding 向量召回 + 特征工程精排 | 准实时（分钟级） | 中 |
| **方案C：实时特征 + 深度学习精排（DIN/DIEN）** | 成熟期（DAU > 500万） | 用户行为序列建模 | 实时（秒级） | 高 |

**当前建议：先做方案A，预留方案B的数据埋点**。

### 推荐链路核心设计

```
用户请求
   ↓
【召回层】多路并行召回（Top N ≈ 500~2000）
   ├── 协同过滤召回
   ├── 热度召回（兜底）
   └── 关键词/标签召回
   ↓
【粗排】轻量模型快速过滤（→ Top 100~200）
   ↓
【精排】主模型打分（CTR/停留时长预估）
   ↓
【重排】业务规则（去重/去广/时效/多样性打散）
   ↓
返回 Top 20~50 给前端
```

## 产品需要补充确认的问题

1. **核心优化目标是什么？** "提升停留时长"和"提升点击率"在模型目标函数上是不同的，请产品明确 1 个主要优化指标和 1~2 个护栏指标。
2. **当前是否已有行为埋点？** 这直接决定项目是"2周能上线"还是"需要先等数据积累1个月"。
3. **推荐场景是哪些？** 首页 Feed、详情页猜你喜欢、搜索结果页，工作量差异可达 2~3 倍。
4. **内容库规模是多少？** 万级和百万级的技术方案选型差异很大。
5. **是否有现成的 A/B 实验平台？** 没有则无法证明推荐系统有效。
6. **用户规模和增长预期？** 决定是否需要从第一天就设计水平扩展能力。

## 测试用例 2：开发 → 产品

**接口**：`POST /api/translate`
**方向**：`dev_to_pm`
**状态**：✅ 通过

### 输入

```
我们对核心接口进行了异步化重构，将同步阻塞调用改为基于 asyncio 的协程模型，
并引入了连接池复用机制，QPS 从 800 提升到 3200，P99 延迟从 1200ms 降到 180ms。
```

### 翻译输出

## 一句话总结

我们对服务器的"接待能力"进行了彻底升级——同等硬件资源下，系统可同时处理的用户请求量**提升了4倍**，用户等待响应的最长时间从**1.2秒压缩到0.18秒**，高峰期卡顿问题基本消除。

## 用户体验影响

**P99延迟**：100个用户里等待最久的那1个人，之前等1.2秒，现在只需0.18秒。

| 场景 | 改善前 | 改善后 |
|---|---|---|
| 用户提交订单/表单 | 偶发明显卡顿，部分用户重复点击 | 响应接近即时，操作确认感强 |
| 大促高峰 | QPS超800后系统排队，体验快速劣化 | 承载上限提升至3200，常规大促不再是瓶颈 |
| 移动端弱网 | 服务端1.2s+网络延迟叠加，用户放弃等待 | 服务端0.18s，整体体验显著改善 |

## 业务价值

- **流量损失降低**：P99 从1200ms→180ms，此前因超时流失的用户请求现在可以被有效承接
- **大促容量**：同样的服务器支撑3200 QPS，相当于**节省了3倍扩容成本**
- **转化率**：高峰期接口超时导致的订单失败率预期下降

## 解锁的新产品可能性

1. 实时搜索建议、即时消息通知等对延迟敏感的功能可以重新评估排期
2. 秒杀、限时抢购类活动可以支撑更大规模
3. 可以同时运行更多 A/B 实验分组

## 建议的产品跟进动作

1. **【立即】** 与数据团队确认监控看板，重点追踪支付/提交成功率变化
2. **【本周】** 盘点被"系统性能"卡住的需求池，重新评估优先级
3. **【本月】** 与运营重新核定大促活动规模的安全阈值
4. **【下季度】** 将 API 开放/第三方联动需求提上议程
