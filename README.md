# 职能沟通翻译助手

> 帮助产品经理和开发工程师跨越"语言鸿沟"，让需求沟通更高效。

---

## 功能说明

### 这个工具做什么？

在企业协作中，产品经理和开发工程师因为思维模式不同，经常出现理解偏差。本工具利用 AI 实现**双向沟通翻译**：

- **产品 → 开发**：将模糊的需求描述翻译为技术语言，包含技术方案建议、工作量拆解、需要确认的问题
- **开发 → 产品**：将技术指标翻译为业务影响，挖掘商业价值，给出可执行的产品建议
- **技术/产品 → 运营管理**：将内容翻译为管理层关注的业务指标、ROI 和决策建议

### 核心功能

| 功能 | 说明 |
|---|---|
| 三向翻译 | 产品→开发、开发→产品、内容→运营管理层 |
| 流式输出 | 翻译结果实时逐字生成，无需等待 |
| 智能识别 | 自动判断输入属于哪种视角并推荐翻译方向 |
| 主动补充 | 每次翻译都会提出对方视角缺失的关键问题 |
| 在线配置 | 支持在界面中配置 Base URL、Token、Model，无需重启 |

## 快速开始

### 1. 安装依赖

```bash
git clone https://github.com/JimMir45/communication-translator.git
cd communication-translator
pip install -r requirements.txt
```

### 2. 配置 API Key

```bash
cp .env.example .env
```

编辑 `.env` 文件，填入你的配置：

```env
ANTHROPIC_BASE_URL=https://api.anthropic.com
ANTHROPIC_AUTH_TOKEN=your_api_key_here
ANTHROPIC_MODEL=claude-opus-4-6
```

### 3. 运行项目

```bash
python main.py
```

浏览器访问：**http://localhost:8000**

> 也可以用 uvicorn 启动（支持热重载）：
> ```bash
> uvicorn main:app --reload --port 8000
> ```

---

## 在线配置模型

点击右上角 ⚙️ 图标，可以在界面中动态配置：

- **Base URL**：API 接入地址（支持代理/私有化部署）
- **API Token**：鉴权密钥
- **Model**：模型名称（如 `claude-opus-4-6`、`claude-sonnet-4-5`）

配置保存在浏览器 `localStorage`，优先级高于服务器 `.env` 配置，留空则自动使用服务器默认值。

## 测试用例

### 用例 1：产品 → 开发

**输入（产品视角）：**
```
我们需要一个智能推荐功能，提升用户停留时长。
希望根据用户历史行为，给他推荐他感兴趣的内容，让用户在平台上多待一会。
```

**翻译输出（节选）：**

> **需求本质**：构建一套基于用户行为数据的个性化内容推荐系统，核心目标是提升用户会话时长和内容消费深度。
>
> **技术方案**：推荐算法选型建议协同过滤（冷启动阶段）+ 实时行为序列模型（成熟阶段）...
>
> **需要产品确认的问题**：
> 1. 推荐的内容类型范围是什么？是否包含付费内容？
> 2. 用户可以主动标记"不感兴趣"吗？反馈信号如何处理？
> 3. 新用户冷启动期间，推荐策略是什么？

---

### 用例 2：开发 → 产品

**输入（开发视角）：**
```
我们对核心接口进行了异步化重构，将同步阻塞调用改为基于 asyncio 的协程模型，
并引入了连接池复用机制，QPS 从 800 提升到 3200，P99 延迟从 1200ms 降到 180ms。
```

**翻译输出（节选）：**

> **一句话总结**：我们对系统"高速公路"进行了扩容改造——原来同一时间只能跑 800 辆车，现在能跑 4000 辆，而且堵车等待时间从平均 1.2 秒缩短到 0.18 秒。
>
> **用户体验影响**：高峰期页面加载速度显著提升，原来偶发的"转圈"卡顿现象将基本消除...
>
> **解锁的新产品可能性**：
> - 可以支持更高并发的限时秒杀/抢购活动
> - 可以推进实时消息、直播互动等对延迟敏感的功能

## 提示词设计说明

### 核心设计思路

本项目的提示词工程基于一个核心洞察：**产品经理和开发工程师的沟通问题，本质上是两种"世界观"的碰撞，而不仅仅是术语差异。**

因此，提示词的设计不是"帮我改写这段话"，而是**让 AI 真正切换思维框架**。

### 关键设计点

**1. 显性化思维模式差异**

在每个方向的 system prompt 中，明确定义了两种角色的优先级排序：

```
产品经理的世界：用户价值 → 商业目标 → 功能描述（往往忽略技术边界）
开发工程师的世界：数据结构 → 接口设计 → 实现方案 → 工作量 → 风险
```

这不是装饰性文字，而是让模型在生成每一段内容时都有明确的视角参照系。

**2. 强制结构化输出**

每个翻译方向都有固定的输出章节（而不是自由散文）。好处：
- 倒逼 AI 完整思考每个维度，不会因为输入简单就输出简单
- 阅读者可以直接跳到关注的章节，提升实用性

**3. 主动补充机制**

每个方向的 prompt 中都有"主动提出对方视角缺失的问题"的要求。这是产品实用价值的核心——翻译的目的不只是"让对方听懂"，更是"帮对方想到他没想到的"。

**4. 量化约束**

要求将模糊描述转化为可量化指标：
- "做快一点" → "P99 延迟 < 200ms"
- "流量很大" → "预估 QPS 区间"

**5. 角色身份设定**

不同方向使用不同的角色身份（"资深全栈工程师"、"产品总监"、"业务运营专家"），而不是统一用一个"助手"角色，增强翻译结果的专业深度。

## 技术架构

```
Frontend (index.html)
    │  SSE / fetch streaming
    ▼
FastAPI Backend (main.py)
    │  Anthropic Python SDK
    ▼
Claude API (configurable endpoint)
    │
prompts.py  ──  Prompt engineering layer
```

**技术栈：**
- 后端：Python 3.9+, FastAPI, Anthropic SDK
- 前端：原生 HTML/CSS/JS + marked.js（Markdown 渲染）
- 流式传输：Server-Sent Events (SSE)
